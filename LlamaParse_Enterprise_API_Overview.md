A few things I'd still pressure-test before you let it build:
1. The confirm_count ≥ 3 threshold is a guess — and it has a cold-start cost. Until an LLM-detected issuer has been seen 3 times, every occurrence pays the full Tier B cost and never short-circuits. For issuers that appear once a month, that's three months before Tier A ever helps them. Given your volume (one or two seller reports per deal per month), a lot of your ~250 deals may rarely or never reach 3. Consider whether ≥ 2, or "1 human confirmation OR 3 LLM," is more realistic. This is tunable, but worth setting deliberately rather than defaulting to 3.
2. Registry key normalization is doing a lot of silent work. Tier A matches on (normalized_name, trust_pattern). The entire poisoning guard and margin logic is only as good as that normalization — and naming drift is your documented edge case. If "BMW Auto Owner Trust 2024-1" and "BMW AOT 2024-1" normalize differently, you get duplicate registry rows that each independently climb toward confirm_count = 3, and the margin check compares against the wrong population. I'd want to see the normalization function specified explicitly, not left implicit, because it's load-bearing for everything in Tier A.
3. Adaptive window "widen to 2x and retry once" doubles LLM cost on exactly the hard cases. That's acceptable, but make sure the retry only fires below threshold and there's a cap, so a pathological doc can't trigger repeated widening. Reading the spec, it looks bounded to one retry — just confirm that in the implementation, since "retry once" in a prompt and "retry once" in code sometimes diverge.
4. What happens when Tier B returns an issuer that resolves to a deal but you can't resolve to a specific template? The schema has template_path: str | None. So None is a valid outcome — issuer identified, template unknown (e.g. a new series of a known issuer). Make sure the pipeline handles template_path = None as its own state: you know the issuer, but you've hit the new-deal/new-template gap Chander still owes you the business flow for. That shouldn't crash the swap; it should route to the template-creation path. Confirm that branch exists.
5. Minor: IssuerDetectionResult has both a top-level issuer/servicer/trustee/depositor and the EntityMatch has a role field. There's some redundancy — the role is encoded both by which slot the entity sits in and by its own role string. Not wrong, but make sure they can't disagree (an EntityMatch with role="servicer" sitting in the issuer slot). One source of truth is cleaner.
On the file plan in image 1 — splitting IssuerDetector (core/classification), the Pydantic models (contracts), the registry DDL + CRUD (core/storage), config settings, and the AppContext wiring is the right separation. The "replace inline detect_issuer" line at the bottom confirms it's superseding your earlier runner.py logic rather than bolting on beside it, which is what you want.
Net: this is buildable as-is and it's a genuine improvement over the keyword-first version. The two I'd actually resolve before generating code are the normalization function (#2 — it's silently load-bearing) and the template_path = None branch (#4 — it's where this design meets the unsolved new-deal gap). The confirm_count threshold (#1) you can ship and tune later from real data.
One question that's worth raising with Chander rather than solving in code: does the production DB already store the full trust/series name per active deal? Because if it does, your Tier B "extract the full trust/series name" is partly redundant with a DB lookup — you might be able to match the extracted trust name directly against the deal table and skip a lot of the disambiguation. That ties back to whether the deal ID is knowable at upload, which is still your open question for Tuesday.
