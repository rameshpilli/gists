**Context / diagnosis (don't re-debug this):**

The 504 is an *output*-generation problem, not an extraction problem. LlamaParse and field extraction work. The gateway closes the socket at ~120s because the LLM is streaming a ~9,450-token response, and the bulk of that is narrative fields (e.g. `exec_summary_background`) being returned as full multi-page text inside JSON values — the model echoing the document back to us. We already have that text byte-exact in the raw LlamaParse output. So we are duplicating work and timing out on the duplicate. The earlier 4096-token cap on the agentorchestrator client is already fixed; do not change that further.

**Do NOT:**
- Treat "bump the gateway timeout" as the fix. Raise it with infra in parallel, but build as if 120s is permanent.
- Truncate or drop any field or any narrative content.
- Ask the LLM to return narrative/freestyle prose as field values.

**Build, in this order:**

1. **Split the output into two channels.** The LLM extraction call returns **structured fields only — all 63 — as compact values** (names, IDs, dates, amounts, short categoricals). Narrative / freestyle / `all_data` content is **served from the raw LlamaParse output**, not regenerated by the LLM. Nothing is lost; the freestyle download comes from the raw tab in full.

2. **For any narrative field the user wants located** (not reproduced), have the LLM return a **locator** — page number, heading anchor, or character offset into the raw text — instead of the prose. The UI/raw tab resolves the locator to the actual text.

3. **Make extraction async.** Submit job → run on the decoupled service → report status to the DB → consumer polls or gets a callback → results fetched when ready. No long-held synchronous HTTP request for the gateway to time out. Keep the existing consumer contract (IDPass/Solace) — this is a sync-vs-async change inside it, not a contract change. Confirm the submit/callback shape against how IDPass already routes over Solace.

4. **Validate it's now one call under budget.** With narrative text out of the output, the response for a 39-page doc should drop from ~9,450 tokens to under ~2,000 and finish in ~30–40s. Keep it as a single LLM call.

5. **Multi-pass only as a fallback,** and only if step 4 still lands near the wall for genuinely high field counts. If so: split fields by **output volume**, run the calls **in parallel** (each sees full text as input, emits a partition of fields), merge, validate against schema, retry only failed partitions. Do not default to sequential per-group calls — that makes wall-clock worse.

**Surface back to me, don't guess:**
- Is the 120s an *idle* timeout or a hard wall-clock cap? (If idle, streaming/SSE may sidestep it; if hard, async is the only clean path — which we're building anyway.)
- Confirm the async submit/callback contract that IDPass expects over Solace before wiring it.

**Acceptance:** user downloads all 68 structured fields (compact JSON) plus complete freestyle content (from raw), no truncation, single doc completes without a 504, and nothing in the consumer contract changes.
