Changes / improvements I'd make
1. Tier 1's auto-grow registry needs a poisoning guard. Every confirmed LLM detection and human correction adds a row — good for coverage, but it means a wrong LLM detection that a human didn't catch can write a bad pattern that then short-circuits tier 1 forever after. Two safeguards: only auto-promote a pattern to tier 1 after it's been confirmed N times (or by a human specifically, not just "LLM was confident"), and keep the evidence snippet + source on each row so a bad pattern is auditable and reversible. Distinguish "human-confirmed" rows from "LLM-asserted" rows — they shouldn't have equal authority.
2. 4,000 chars + filename is better than 2,000, but still make the window adaptive. The filename addition is a real improvement (often the cleanest signal). But keep the same caveat as before: parser variance between LlamaParse and LiteParse means "first 4,000 chars" isn't a stable slice, and issuer naming can drift mid-doc. If tier 1 misses and tier 2 also comes back low-confidence, consider widening the scan before going to human. Make 4,000 a config constant, not a literal.
3. Tier 1 "most pattern hits win" still has the cross-contamination problem. A GM doc that references BMW as a comparison can match BMW patterns. Since tier 1 is now your instant cache and skips the LLM entirely, a false tier-1 hit never gets the disambiguation that tier 2 would provide. Mitigation: require tier 1 matches to clear a margin (best match must beat second-best by some gap), and weight filename/header-zone hits above body-text hits. If it's ambiguous, fall through to tier 2 rather than picking.
4. Make tier 2's typed JSON return all four entities, not just the issuer. Since you're already disambiguating issuer/servicer/trustee/depositor, return all of them in the JSON, not just the winner. Reasons: the servicer and trustee are useful downstream (and for the human reviewer to sanity-check), and storing them makes future disambiguation richer. The typed schema should be something like: issuer, servicer, trustee, depositor, each with value + evidence snippet, plus an overall confidence and decision_path.
5. Add a verification cross-check against the DB. This connects to the SQL connector the architecture review centered on. Once tier 1 or tier 2 picks an issuer, validate it against the active-deals mapping from the production DB. If the detected issuer isn't in the active deal list, that's a strong signal something's wrong — route to human even if confidence was high. The DB is your ground truth; the detector should defer to it, not operate independently of it.
6. Tier 3's correction should be cheap for the human. You've got "best guess prefilled" — good. Make sure the human is choosing from the known active deals/issuers (a constrained pick from the DB), not free-typing, so corrections are clean registry rows and not new naming-drift variants. The Hackathon UI everyone liked showed why it guessed (hover-to-explain) — surface tier 2's evidence snippet here so the human can confirm quickly.
Where it sits relative to the rest of the pipeline
This replaces/becomes the issuer-detection step that feeds the doc-class swap. Confirm the contract out of tier 3 is: resolved deal/template identity + confidence + which tier decided + evidence. That object is what swaps the rules (per-pool, not just BMW.xlsx) and what the front end displays as a visible step with confidence — consistent with how extraction and mapping confidence are shown.
Net: the design is correct and the entity-disambiguation is a genuinely good catch. The five things I'd add before building: a registry poisoning guard, a margin requirement on tier 1, returning all four entities from tier 2, a DB cross-check against active deals, and resolving to deal/pool rather than bare issuer. None of those change the three-tier shape — they harden it.
