
Get narrative content out of the LLM output path entirely. The extraction call returns structured fields as compact values only. Long-form/narrative content is served from the raw parse, not regenerated by the model. If you need the LLM to locate a narrative section, have it return a page/heading anchor or character offsets — not the prose. This alone likely drops your response from ~9,450 tokens to under ~2,000, finishing in 30–40s, one call, no timeout change needed. This is exactly your designed 2×2 output model (long-form with coordinates, not long-form reproduced). You just haven't wired the narrative path to honor it yet.
Make the timeout ask in parallel, but don't depend on it. It's a shared corporate gateway — the bump may not be granted, and even at 300s you've built a 140s+/doc synchronous path that won't scale for CLM bulk volumes. Treat extra headroom as nice-to-have, not the fix.
Multi-pass only if step 1 still leaves you near the wall (genuinely many structured fields, some with moderate text). If so, split by output volume, not arbitrary "sections," and run the calls in parallel so wall-clock doesn't balloon — each call sees the full text as input and emits a partition of the fields. The other agent's 5 sequential calls with per-group retry over-engineers it and can make total latency worse.
